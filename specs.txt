Project description
Build a LAN-only “LLM query router” that routes chat requests to nearby trusted machines running local open-source models (via LMStudio), to reduce reliance on remote datacenters. The client chooses the model explicitly (no auto model selection). The router never sees prompt content; it only uses metadata + server stats to pick a compute server. Compute servers run a lightweight sidecar agent that reports capabilities and live load, enforces concurrency, and proxies requests to LMStudio.

Components, responsibilities, and contracts
1) React Client (UI + client networking)
Responsibilities
ChatGPT-like UI: message list, input box, model dropdown.


Maintain conversation history client-side and include it in each /run.
Assume that we already know the IP address/port of the router


For each user send:


generate request_id (UUID) once


compute prompt_chars (and optionally estimated_prompt_tokens = ceil(chars/4))


choose max_tokens (fixed default is fine for demo)


call router /route (metadata only)


call compute agent /run (full request)


Reliability:


timeouts + AbortController cancel


at most 1 retry on /run using the same request_id


if compute endpoint is unreachable, re-call /route and try another server (still same request_id)


What it must NOT do
No streaming.


No server-side state assumptions (history stays client-side).


No tokenizers required (chars/4 estimate is fine).


Contracts it uses
POST /route on router (RouteRequest → RouteResponse)


POST /run on compute agent (RunRequest → RunResponse)



2) Routing Server (Go)
Responsibilities
Maintain registry of compute servers and their static capabilities.


Maintain dynamic state from heartbeats (in_flight, status, ema_tps, last_seen).


Decide routing for each incoming metadata request.


State it keeps
capabilities[server_id] from /register


state[server_id] from /heartbeat (+ derived DOWN if stale)


Failure detection
Mark a server DOWN if it misses heartbeats for ~3 * interval + grace (demo default: heartbeat every 2s, down after ~7s).


Exclude DOWN servers from routing.


Scheduling policy (v0)
Filter to servers that:


are registered


support model_id


are not DOWN


status != DRAINING


Choose the server with:


lowest in_flight


tie-break by estimated runtime:


estimated_runtime ≈ (estimated_prompt_tokens + max_tokens) / max(ema_tps, fallback_tps)


if estimated_prompt_tokens missing, use ceil(prompt_chars/4)


fallback_tps constant if ema_tps missing


Contracts it serves
POST /register (compute → router): static capabilities


POST /heartbeat (compute → router): dynamic load


POST /route (client → router): routing decision



3) Compute Agent (Python + FastAPI) running on each compute server
Responsibilities
Proxy execution requests to LMStudio.


Enforce availability/concurrency (LLM servers are “single-lane bridges”):


demo default: max_concurrency = 1


track in_flight


if at capacity and no queue: return 429 SERVER_BUSY


Provide idempotency so client retries don’t double-run:


cache request_id -> full RunResponse for ~2–5 minutes


duplicates return cached response (optionally cached: true)


Measure performance and report dynamic load:


measure elapsed_ms for each job


compute throughput (tokens/sec) if possible:


best: parse LMStudio response usage.completion_tokens


fallback: estimate completion tokens as ceil(len(output_text)/4)


update ema_tps (α ~ 0.2; ignore tiny jobs to reduce noise)


On startup:


call router /register once with capabilities


Continuously:


send router /heartbeat every ~2 seconds with dynamic stats


Availability definition
READY if in_flight < max_concurrency


BUSY if in_flight == max_concurrency


DRAINING if refusing new work (manual switch for demo control)


Router infers DOWN when heartbeats stop.


Contracts it serves
POST /run (client → compute): full request, non-streaming response


Contracts it calls
POST /register (compute → router)


POST /heartbeat (compute → router)



4) LMStudio (local model host)
Responsibilities
Actually run the model.


Expose an HTTP API that the compute agent proxies to.


The compute agent is responsible for:


translating your simplified schema to LMStudio’s expected format


extracting usage metrics if provided


Important constraint
Don’t rely on LMStudio for stats/availability; treat it as the “engine” only. The sidecar agent is your control plane.



Shared contracts (the schema you all code against)
Router-facing metadata (client → router /route)
Required:
request_id (UUID)


timestamp_ms


model_id


prompt_chars


max_tokens
 Optional:


estimated_prompt_tokens (else router estimates from chars)


Compute capability (compute → router /register)
Required:
server_id


endpoint (compute agent base URL)


models (array of model_id)


max_concurrency


device_type (CPU/GPU)
 Optional:


context_limit


vram_gb


Heartbeat (compute → router /heartbeat)
Required:
server_id


timestamp_ms


status (READY/BUSY/DRAINING)


in_flight
 Optional:


ema_tps


last_job_ms


Full run (client → compute /run)
Required:
request_id


timestamp_ms


model_id


max_tokens


messages[] with {role: system|user|assistant, content: string}


Response:
request_id, model_id


output: {role: assistant, content}


timing.elapsed_ms


usage (real if available, else estimated/nulls)


optional cached: true if served from idempotency cache



Default timeouts and retry rules (client-side)
Router /route timeout: ~1s


Compute /run connect timeout: ~500ms


Compute /run overall timeout: ~30s (optionally 60s for big models)


Retry: 1 retry max on /run with the same request_id


If endpoint unreachable: re-route and try another server



Recommended build plan so everyone can work independently
Router engineer (Go)
Implement /register, /heartbeat, /route


In-memory maps for capabilities/state


DOWN detection by missed heartbeats


Routing score function (least in_flight + runtime tie-break)


Compute agent engineer (FastAPI)
Implement /run with concurrency gate + idempotency cache


Proxy to LMStudio


Measure elapsed_ms


Compute token estimate and EMA TPS


Implement register-on-start and periodic heartbeat loop


Frontend engineer (React)
Chat UI + model dropdown


Implement send flow: /route → /run


Timeouts + cancel + 1 retry


Display: server chosen + latency (nice demo polish)



If you want one extra “alignment booster”: agree on a canonical model_id list up front (even just 2–3 entries) and have each compute agent map those to its local LMStudio model names. That prevents the “same model, different string” bug that kills demos.
